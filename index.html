Secure Interoperability Platform

Follow a data request
from origin to immutable audit

ASCII Road is a neutral middleware network that lets institutions securely share citizen data — only with the citizen's explicit, cryptographic consent. No institution ever sees data it wasn't authorized for. No request can be denied or falsified after the fact.

The core problem
Institutions need citizen data from each other. Today this happens via fax, email, or insecure APIs with no audit trail.
The ASCII Road solution
A consent-first, cryptographically-verified data exchange where every action is signed, logged, and provable forever.
Key design principle
The Orchestrator is a referee, not a middleman. Actual data flows directly between institutions — never through the Orchestrator.

🚀 Phase 1 — Initiation

Institution A sends an official, signed request

💡 Think of it like a notarized legal letter — it carries a cryptographic stamp that proves exactly who sent it and that it hasn't been tampered with since.
1

The SDK creates a signed request

Institution A's system (e.g., a bank) uses the ASCII Road SDK to build a request that says: "We want Citizen X's Tax Record, here's why." Before sending, the SDK signs this request with the bank's private key — like a wax seal that proves it came from them.

⚡ Private keys never leave the institution's server
2

Orchestrator verifies the sender's identity

The Orchestrator receives the request and fetches Institution A's registered public key from its database. It uses this to verify the signature — confirming the request genuinely came from Institution A and was not modified in transit.

3

RBAC check — is this institution allowed to ask?

Not every institution can request every type of data. A Worker Pod checks a Role-Based Access Control (RBAC) table: does Institution A's role grant it access to Tax Records from Institution B? If not — the request is rejected here, before the citizen is ever notified.

⚡ Zero citizen notification for unauthorized requests
4

HMAC routing — finding the citizen, without storing their real ID

The Orchestrator never stores plain-text citizen IDs in its database. Instead, it applies an HMAC-SHA256 hash (using a secret key that never leaves the server's memory) to produce an anonymous lookup ID. This is used to find which institution holds the citizen's data — and where to send the request.

⚡ Database contains zero reversible citizen identifiers
5

Asynchronous response — connection closes immediately

The Orchestrator responds with 202 Accepted and a Request ID. The connection closes. Institution A's SDK waits for an async webhook callback — it does not hold an open connection. The Orchestrator now handles everything in the background.

🔔 Phase 2 — Citizen Consent

The citizen is asked, and must sign their own approval

💡 Like signing a physical document in front of a notary — except the "notary" is your phone's secure hardware chip, and the signature is mathematically impossible to forge.
1

Push notification delivered to citizen's phone

The Orchestrator triggers a push notification via FCM (Android) or Apple Push Notification Service. The message is specific: "Bank A requests your Tax Assessment record for loan processing." The citizen sees exactly who is asking, what data, and why.

2

Consent screen — full transparency

The citizen opens the app and sees a structured consent screen showing: (1) the requesting institution, (2) the exact data schema requested, (3) the stated reason. They can tap Approve or Deny. Nothing happens without explicit action.

3

The Secure Enclave signs the approval

When the citizen taps Approve, the app calls signConsent() — this uses the device's Secure Enclave (a physically isolated security chip in every modern iPhone and Android). The Secure Enclave signs the consent with a key that never leaves the chip, even from the operating system. Biometric authentication (Face ID / fingerprint) is required to authorize this signing.

⚡ This signature proves the REAL citizen approved it — not a hacker, not a middleman
4

Signed approval posted to Orchestrator

The mobile app sends POST /api/v1/consent with the Request ID and the cryptographic signature. The API responds 200 OK immediately and closes the connection — processing continues asynchronously in the background via Kafka.

🔑 Phase 3 — Credential Issuance

A 5-minute, one-use access pass is issued to Institution A

💡 Like a time-locked hotel key card — it only works for one room (one data type), it expires in 5 minutes, and it was issued by the hotel's front desk (the Orchestrator) only after verifying your booking (the citizen's consent).
1

Worker verifies the citizen's signature

A Worker Pod consumes the consent.received Kafka event. It fetches the citizen's registered public key from the database and calls verifyCitizenSignature(). If the signature is invalid — the process stops. Consent was forged or tampered with.

2

JWT credential generated

Consent verified. The Worker generates a JWT (JSON Web Token) — a compact, self-contained credential. It is signed with the Orchestrator's own private key. The JWT encodes: who the citizen is (as an anonymous hashed ID), which institution may use it, which data schema it unlocks, and when it expires (5 minutes from now).

⚡ The JWT contains NO plain-text citizen information
3

Delivered to Institution A via webhook

The Orchestrator sends an HTTP POST to Institution A's webhook endpoint, delivering the JWT alongside the Provider's API URL. Institution A's SDK receives this, verifies the JWT, and is now ready to make the direct data fetch. The clock is ticking — 5 minutes to use it.

🔒 Phase 4 — Peer-to-Peer Transfer

Data flows directly — the Orchestrator is not in this path

💡 The Orchestrator already verified everything and issued the key. Now Institution A walks directly to Institution B's vault, shows the key, and the vault hands over exactly what was authorized — nothing more, cryptographically signed.
1

mTLS handshake — mutual identity verification

mTLS (Mutual TLS) means both sides prove who they are before any data flows. Institution A presents its X.509 certificate. Institution B presents its certificate. Both are validated against the ASCII Road Root Certificate Authority. If either certificate is invalid — connection refused. This prevents impersonation attacks.

⚡ Unlike regular HTTPS, both parties must authenticate — not just the server
2

Hono Middleware intercepts and validates the JWT

The Hono Middleware (running in Institution B's infrastructure) intercepts all incoming requests. It fetches the Orchestrator's public key from a cached store, then verifies the JWT signature. If the signature doesn't match — 401 Unauthorized. No data is served.

It then runs three checks: Expiration (is it within 5 minutes?), Identity (does req match Institution A?), Scope (is schema_slug authorized for this endpoint?).

3

Data fetched using anonymous citizen ID

The middleware calls fetchRecord(hashedId) — using the hashed citizen ID from the JWT. Institution B's internal database looks up the record using this hash. The raw citizen name/national ID is never exposed to the Orchestrator or in transit.

4

Schema normalization and digital signing

mapSchema() transforms Institution B's internal data format to the ASCII Road standard format. Then signPayload() computes a SHA-256 hash of the data and signs it with Institution B's private key. This signature travels with the data.

⚡ Institution A can prove the data came from B unmodified — forever

JWT Claims Verification Logic
// 1. Verify signature came from Orchestrator
valid = verify(jwt, orchestratorPublicKey)
if (!valid) return 401 Unauthorized

// 2. Check expiry (max 5 minutes)
if (jwt.exp < Date.now()) return 401 Expired

// 3. Check requester identity
if (jwt.req !== clientCertificate.institutionId) return 403

// 4. Check data scope
if (!endpoint.allows(jwt.scope)) return 403 Forbidden

// All checks pass → fetch data
fetchRecord(jwt.sub) // sub = hashed citizen ID

📋 Phase 5 — Audit & Finality

A tamper-proof, permanent record is written — provable forever

💡 Like a notarized receipt in a fireproof safe — the receipt contains a cryptographic fingerprint of exactly what was transferred. If anyone ever claims the data was different, or that the transfer didn't happen, the audit log proves the truth mathematically.
1

Institution A generates a signed receipt

After receiving the data, Institution A's SDK verifies Institution B's signature, then computes a SHA-256 hash of the data payload — a unique "fingerprint" of exactly what was received. It bundles this hash with the Provider's signature into a Receipt object, then signs the receipt with its own institution key.

⚡ The receipt is a cryptographic proof of the exact data transferred
2

Receipt posted to the Orchestrator

The SDK posts POST /api/v1/receipt with the signed receipt. The Orchestrator verifies the requester's signature, then immediately publishes the receipt data to a special append-only Kafka stream — the immutable audit log.

3

Immutable Kafka ledger — KRaft consensus

Kafka in KRaft mode means the log is managed by a distributed consensus protocol. Once an event is written to this topic, it cannot be deleted or modified — not even by a system administrator. The topic is configured as append-only with infinite retention. This is the "blockchain" equivalent — but practical and proven at scale.

4

Audit Worker finalizes the database record

An Audit Ledger Worker consumes the Kafka event and creates a permanent DataRequestSignature record in PostgreSQL: type PROVIDER, the SHA-256 data hash, and the Provider's signature. The DataRequest status is updated to COMPLETED. The lifecycle is closed.

⚡ The transfer is now mathematically provable in perpetuity
